# 遊戲企劃與產品需求文件 (PRD)：互動對話角色扮演遊戲 (Dialogue Roleplay Game)

## 1. 專案概述 (Project Overview)
**產品經理 (TPM) 視角：打造多情境、多時態的語境推理解謎遊戲**

*   **遊戲名稱**：Time-Traveling Detective & Reporter (時空偵探與記者) / 暫定
*   **設計理念**：將單純的文法問答升級為「收集線索 ➡️ 統整匯報」的雙階段任務。透過「找嫌犯」或「科普現象調查」的包裝，讓學生自然而然地運用不同時態的疑問句與直述句。

### A. 學習目標 (Learning Objectives)
1.  熟悉各種生活情境的對話脈絡（如犯罪調查、科學現象採訪）。
2.  訓練肯定與否定的疑問句、肯定與否定之簡答與詳答。
3.  訓練直述句（包含過去、現在、未來、現在進行等四個時態的陳述能力）。

---

## 2. 核心玩法 (Core Gameplay)

**情境設定 (Scenarios)**：
*   **找嫌犯模式**：玩家化身「偵探」，為了找出嫌疑犯而進行調查。
*   **科普文章模式**：玩家化身「記者」，為了寫出一篇科普報導而採訪專家。

### Phase 1: 提問階段 (Make a Question - 收集線索)

*   **考點**：各時態的疑問句和詳答。
*   **流程**：
    1.  **關卡獨立選擇機制**：遊戲大廳會列出該案件的**四個時態調查任務**（例如：過去、現在、未來、現在進行）。玩家可以**自由選擇、獨立點選**進入任何一個時態進行調查，且隨時可以中途離開關卡回到大廳，各時態之間並非強制連續的線性關係。
    2.  玩家進入單一時態關卡後，需針對畫面給予的提示（Prompt），向 NPC 提問以獲取線索。
    3.  **雙重解鎖機制 (Scaffolding)**：單一關卡內的線索分為「動詞」與「時間/頻率」兩層。請注意，**各個時態都必須配有其專屬的動詞變化與時間/頻率提示詞**（例如：過去式會出現 yesterday, last night；現在進行式會出現 now, right now 等），以下僅為舉例：
        *   *Step 1 (動詞提示)*：畫面依據當前關卡的時態出現對應的動詞提示（例：`eat, drink, walk`）。玩家需**自行輸入 (打字)** 完整的疑問句問出動作（例：過去式關卡為 *Did he eat?* / 現在式關卡為 *Does he drink?*）。送出後系統會判斷此線索是否符合嫌犯特徵，NPC 以「語音＋對話泡泡文字」回答後解鎖下一步。此階段的提問只會包含助動詞 (Do/Does/Did/Will/Be 等)、主詞及動詞。時間資訊會出現在下個階段，而 Wh- 問句將會被安排在未來的其他遊戲模組中。
        *   *Step 2 (時間/頻率提示)*：畫面出現與該時態相符的時間或頻率副詞（例：`every night, usually, always` 或 `yesterday`, `tomorrow` 等）。玩家需**自行輸入 (打字)** 將時間/頻率詞加入問句中提問（例：*Does he often eat breakfast?* / *Did they watch a movie yesterday?* / *Will she walk to school tomorrow?*）。請注意，題庫設計與判定時必須確保句中包含受詞，且須檢查句子的語意合理性。送出後，NPC 以「語音＋對話泡泡文字」回答，若成功找到該時態的線索，即可解鎖進入下個時態。
    4.  玩家完成該時態的「動詞＋時間」線索後，該關卡即算通關（獲得該時態的線索碎片），玩家會被導回遊戲大廳，可隨時自由選擇挑戰下一個時態，或退出遊戲。

### Phase 2: 回答階段 (Respond - 統整與匯報)
*   **考點**：各時態的直述句文法。
*   **流程**：
    1.  **找嫌犯遊戲**：
        *   *單人模式*：玩家（偵探）需向警察局長 (NPC) 統整回報。使用者需用直述句說明該嫌疑犯在四個時態下的行為（例：*He ate ramen yesterday. He usually drinks coffee.*）。若文法全對，系統將展示真正的嫌犯圖片，宣布破案。
        *   *多人/小組模式 (Classroom)*：使用者輪流用英文直述句描述自己手中嫌犯在四個時態的特徵，讓其他玩家或小組猜。時間內猜對最多嫌犯圖片的小組獲勝。
    2.  **科普文章遊戲**：
        *   收集完四個時態的科普線索後，玩家輪流分享自己「寫好（拼湊完成）」的科普文章（直述句）。

## 3. UI/UX 設計規範 (UI/UX Design)
**設計師視角：引導式介面與資訊層級呈現**

*   **全英文介面原則 (English Only UI)**：為維持沉浸式英語學習環境，所有遊戲內的任務說明 (Task Prompts)、階段提示 (Step Hints)、錯誤回饋 (Error Feedback) 及 UI 介面，皆須使用全英文呈現，不可出現中翻英的提示。
*   **使用者名稱輸入機制**：使用者名稱需在「每次選擇一個新案件 (Case) 準備開始時」輸入，以確保成績能正確綁定至該次遊玩紀錄，而不是在進入遊戲大廳時輸入。
*   **視覺風格**：
    *   *偵探模式*：強調「美式漫畫」或「復古偵探」風格，包含帶有懸疑感的偵探板 (Detective Board)、紅線、放大鏡、泛黃檔案紙、以及復古打字機字體。
    *   *記者模式*：強調「現代新聞編輯室」風格，包含發佈會講台、麥克風、筆記本、錄音筆。
*   **Phase 1 (提問) 介面配置**：
    *   **進場動畫 (Intro Animation)**：進入遊戲前，先播放一段說明案件發生背景（例如：畫作被盜的瞬間、或是發現神秘腳步聲）的動畫，幫助玩家快速進入角色情境。對話框需放大，且高度固定避免忽大忽小。
    *   **左側/上方 (NPC 互動區)**：顯示受訪者或目擊者，以及他們的對話氣泡。
    *   **右側/下方 (線索板 Tracker)**：顯示四個時態的進度條。目前正在解鎖的時態會亮起，並顯示「鎖頭」圖示的提示單字（如 `eat` 🔒）。
    *   **底部 (操作區)**：提供文字輸入框 (Input Field)，玩家需**自己打字**建構完整的疑問句。送出後系統會判斷並給予回饋，若找到一個時態的線索（包含動詞與時間）即可解鎖進入下個時態。
*   **Phase 2 (回報) 介面配置**：
    *   畫面轉換為「報告書」或「新聞稿」的填空/重組畫面。
    *   玩家需把 Phase 1 收集到的線索碎片，用正確的時態組合合成完整的直述句段落。

## 4. 技術與架構規格 (Tech Lead - Specifications)
**技術主管視角：時態狀態機與多人連線機制**

*   **核心模組 (Core Modules)**：
    1.  `InvestigationEngine`: 管理四個時態的進度 (State Machine)。必須嚴格控制 `Verb Phase -> Time Phase -> Next Tense` 的順序。
    2.  `MultiplayerService`: 針對 Phase 2 的多人模式，若為線上即時互動，需引入 WebSocket (如 Socket.io) 或 Firebase 即時資料庫，處理房間建立 (Room creation)、輪流發言 (Turn-based logic) 與計分板 (Leaderboard)。若為線下課堂使用，則只需前端單機計分邏輯即可。
    3.  `VoiceEngine` (TTS 模組): 採用「文字轉語音 (Text-To-Speech, TTS)」技術（如 Web Speech API 或雲端 TTS 服務），自動將題庫中 NPC 的回答轉換為語音輸出。此方案具備高敏捷性與擴充性，未來題庫更新時無需重新錄製音檔，且可透過參數快速切換不同 NPC 的聲音特質（如性別、語調）。
*   **資料庫與 CSV 結構需求 (Data Schema)**：
    題庫需從原本的「單題問答」改為「案件包 (Case Bundle)」結構。一個 Case 需包含 4 個時態的 Target Questions 與 Target Statements。

## 5. CSV 題庫欄位雛形 (Draft Schema for DB)
未來的 `.csv` 將以「案件 (Case)」為單位：
*   `Case_ID` (案件編號)
*   `Mode` (偵探/科普)
*   `Intro_Story` (英文情境簡介)
*   `BGM_URL` (專屬背景音樂連結)
*   `Tense_1_Verb_Target` (時態1-動詞問句)
*   `Tense_1_Time_Target` (時態1-時間問句)
*   `Tense_1_Statement` (時態1-最終直述句)
*   *(以此類推至 Tense 4)*
*   `Image_URL` (最終嫌犯或科普圖片)
*   `Pronoun` (NPC 角色人稱)

## 6. 平台整合與共用模組 (Platform Integration & Shared Modules)
**TL 視角：無縫接入現有學習平台，最大化重用現有元件**

*   **遊戲入口 (Game Entry)**：
    *   **位置**：整合於「文法大廳 (Grammar Hub)」的「時態 (Tenses)」區塊的最後。
    *   **選單呈現**：點擊進入後，會顯示不同的「案件主題 (Themes / Modes)」（如：找嫌犯、科普調查）。玩家點擊主題後，即載入對應的 CSV 資料並開始遊戲（兩階段解鎖）。
*   **共用數據模組 (Shared Data Tracking)**：
    *   強制使用全站共用的 `GrammarDataTracker` 類別 (位於 `shared/game-core/`)。
    *   **數據寫入**：遊戲結束或中斷時，需呼叫 `tracker.endGame()` 將成績、剩餘生命值、作答耗時寫入 `grammar_platform_data` (LocalStorage)，以利「教師儀表板 (Teacher Dashboard)」統一讀取。
*   **共用 UI 元件 (Shared UI Components)**：
    *   **按鈕與彈窗**：必須引入 `shared/styles/ui-components.scss`，使用平台標準的按鈕 (`.btn-primary`, `.btn-secondary`) 與對話框樣式，確保全站風格一致。
    *   **計分與生命值**：必須使用 `GrammarScoreboard` 模組，統一管理愛心 (Lives) 的扣除與分數 (Score) 的增加。
    *   **本遊戲專屬 UI (Custom UI)**：除了上述共用元件外，本遊戲需獨立開發「案件進度鎖頭 🔒」與「對話氣泡 (Speech Bubble)」等專屬介面元件。
