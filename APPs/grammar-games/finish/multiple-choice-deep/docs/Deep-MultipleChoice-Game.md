# 偵探解謎/密室逃脫 (Detective / Escape Room Choice)

## 1. 遊戲設計師 (Game Designer) 視角
### 1.1 核心玩法
- **主題呈現**：將「長篇閱讀」或「複雜文法題」化身為案發現場的「線索 (Clues)」、「日記」或「密碼鎖 (Combination Lock)」。
- **解謎機制**：時間是靜止的，玩家有充足的時間閱讀、分析句子結構，找出隱藏在文字中的「關鍵密碼」。
- **玩家操作**：傳統的 A、B、C、D 四個選項，被包裝成四把「鑰匙 (Keys)」、「密碼齒輪」或「證物 (Evidence)」。
- **深度思考輔助**：
  - **螢光筆 (Highlighter)**：允許玩家在題目上畫重點（例如標記主詞、動詞），幫助大腦梳理邏輯。
  - **漸進式提示 (Progressive Hints)**：如果玩家思考超過 30 秒，旁邊的「放大鏡」圖示會發光，點擊後會刪除一個錯誤選項（50/50），或高亮題目中的關鍵字。

### 1.2 教育目標
- 訓練玩家的**邏輯推演**、**上下文理解**與**文法結構分析**。
- 適用於：長句填空、閱讀測驗、克漏字、時態語態等需要「深思熟慮」的題型。

## 2. 技術負責人 (Tech Lead) 視角
### 2.1 系統架構與技術棧
- **文字解析 (Text Parsing)**：實作文字高亮（Highlighting）功能，需要將題目字串拆解為可互動的 DOM 元素（Span）。
- **拖曳互動 (Drag & Drop)**：使用 HTML5 Drag and Drop API 或第三方套件（如 `dnd-kit`），實作將「鑰匙」拖曳到「鎖孔」的互動。

### 2.2 核心資料結構 (Data Structure)
```typescript
interface DetectivePuzzleState {
  clues: string[]; // 題目或閱讀測驗的段落
  keys: KeyOption[]; // 四個選項
  hintsRemaining: number; // 剩餘提示次數
  isSolved: boolean; // 是否解開密碼
}

interface KeyOption {
  id: string;
  text: string; // 選項文字
  isCorrect: boolean; // 是否為正確鑰匙
}
```

### 2.3 效能與擴展性
- **組件解耦**：將「線索區 (Clue Board)」、「工具列 (Toolbox)」、「密碼鎖 (Lock Panel)」徹底分離，方便未來替換不同風格的密室場景。

## 3. 技術專案經理 (TPM) 視角
### 3.1 專案開發階段 (Milestones)
- **Phase 1: 解謎核心邏輯與 UI 佈局 (Week 1)**
  - 實作題目顯示、選項（鑰匙）的拖曳或點擊輸入、密碼驗證邏輯。
- **Phase 2: 輔助工具與互動 (Week 2)**
  - 實作螢光筆高亮、提示系統（刪去法、關鍵字標示）。
- **Phase 3: 氛圍營造與音效 (Week 3)**
  - 實作開鎖成功/失敗動畫、背景懸疑音樂、紙張翻閱音效。

### 3.2 風險管理
- **行動裝置適配 (Mobile Responsiveness)**：拖曳操作在手機上容易與滾動（Scroll）衝突。需實作「點擊選取 (Tap to Select)」作為拖曳的替代方案。

---

## 4. UI/UX 設計師視角
### 4.1 視覺層級與排版
- **沉浸式背景**：使用保險箱、上鎖的門、書桌等情境，營造解謎氛圍。
- **線索閱讀區**：題目必須顯示在「羊皮紙」、「筆記本」或「電腦螢幕」的 UI 上，字體清晰易讀。
- **工具列**：將提示按鈕（放大鏡）、螢光筆等工具放置在畫面邊緣，不干擾閱讀。

### 4.2 互動回饋 (Feedback)
- **輸入回饋**：將「鑰匙」插入鎖孔或轉動「密碼齒輪」時，需有清脆的機械音效與輕微的震動（Haptic Feedback）。
- **答錯回饋**：密碼錯誤時，鎖頭發出沉悶的「喀」聲並閃爍紅光，玩家可重新嘗試，不立刻扣除生命值（鼓勵試錯）。
- **答對回饋**：密碼正確時，鎖頭彈開，播放勝利音效，並顯示詳解（案情真相）。
