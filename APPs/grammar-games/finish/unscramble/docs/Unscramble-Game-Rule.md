# 火車車廂重組遊戲設計與開發規格書 (Train Unscramble Game Rule)

本文件由線上英語遊戲設計師、技術負責人 (TL)、技術專案經理 (TPM) 及 UI/UX 設計師共同制定，旨在規範「火車車廂重組遊戲 (Train Unscramble)」的開發標準與設計細節。

---

## 1. 遊戲概念與學習目標 (Game Designer Perspective)

### 1.1 核心目標
- **學習目標**：幫助學習者熟悉英語句子結構（如 S+V+O）、語法規則及單字搭配。
- **目標受眾**：兒童至初階英語學習者（A1-A2）。
- **核心玩法**：將打亂的單字/詞組重新排列成語法正確且語意通順的句子。

### 1.2 遊戲模式
- **火車車廂重組遊戲 (Train Unscramble)**
  - **情境**：玩家扮演列車長，需要將載有單字的「車廂」按照正確順序連接到「火車頭」後方，讓火車順利開動。
  - **特色**：強調線性順序與方向性（由左至右），適合訓練完整句子的語感。

---

## 2. UI/UX 設計規範 (UI/UX Designer Perspective)

### 2.1 視覺設計 (Visual Design)
- **火車頭**：固定在畫面左側或上方，帶有提示符號（如大寫字母提示或標點符號提示），作為句子的起點。
- **車廂**：每個單字為一節車廂，底部有輪子，邊緣有連接軸，視覺上暗示可以前後相連。
- **背景**：鐵軌、風景（隨著過關會切換日夜或場景，營造前進的動態感）。

### 2.2 互動設計 (Interaction Design)
- **拖曳 (Drag & Drop)**：
  - 支援滑鼠拖曳與觸控螢幕滑動。
  - 拖曳時車廂應放大 105% 並加上陰影，以增加浮空感。
- **吸附 (Snapping)**：
  - 當車廂靠近目標區域（火車頭後方或其他車廂後方）時，自動吸附並發出清脆的「喀噠」連接音效。
- **即時回饋 (Feedback)**：
  - **正確**：綠色高光、撒花動畫（Confetti）、火車鳴笛聲與火車順利開動的動畫。
  - **錯誤**：紅色閃爍、左右震動（Shake animation）、低沉的錯誤音效。**扣除一次生命值（愛心），若同一題錯誤達 3 次（愛心耗盡），則會跳出該題的「錯誤解說」**，閱讀後進入下一題。
  - **提示 (Hint)**：點擊提示按鈕，可讓下一個正確的單字車廂微微發光。
- **干擾車廂殘留處理 (Distractor Handling)**：
  - 若題目包含干擾項 (`Distractor_Parts`)，當玩家成功拼出完整句子並送出 (Submit) 或火車開動時，剩餘未使用的干擾車廂應**變成半透明灰色**或**被火車的煙霧特效吹飛**，避免玩家困惑是否少拼了單字。

## 3. 技術架構與實作 (Tech Lead Perspective)

### 3.1 前端技術選型
- **框架**：React.js 或 Vue.js (視專案主架構而定)。
- **拖曳套件**：推薦使用 `dnd-kit` 或 `react-beautiful-dnd`（支援無障礙與觸控）。
- **動畫庫**：`Framer Motion` 或 `GSAP`（處理平滑過渡、錯誤震動與火車開動特效）。

### 3.2 資料結構 (Data Structure)
```json
{
  "questionId": "q_001",
  "type": "train_unscramble",
  "originalSentence": "I am playing baseball.",
  "scrambledWords": [
    { "id": "w1", "text": "baseball." },
    { "id": "w2", "text": "I" },
    { "id": "w3", "text": "playing" },
    { "id": "w4", "text": "am" }
  ],
  "correctOrderIds": ["w2", "w4", "w3", "w1"],
  "hints": ["Subject comes first.", "Look for the 'be' verb."]
}
```

### 3.3 核心邏輯
1. **洗牌演算法**：確保每次載入時，單字車廂的順序都是隨機的（避免與正確答案完全一致）。
2. **驗證機制**：
   - 支援多重解答（例如時間副詞放句首或句尾皆可）。
   - 實時驗證（每放一個就檢查）或 提交驗證（全部排完按 Submit）。建議採用**提交驗證**以降低挫折感。

## 4. 專案管理與開發里程碑 (TPM Perspective)

### 4.1 開發階段劃分
- **Phase 1: 核心機制 (Week 1)**
  - 完成基礎 Drag & Drop 功能。
  - 定義 JSON 資料結構與 API 串接格式。
- **Phase 2: 視覺與動畫 (Week 2)**
  - 套用火車頭與車廂的 UI 素材。
  - 實作吸附動畫、錯誤震動與火車開動特效。
- **Phase 3: 音效與優化 (Week 3)**
  - 加入互動音效（拖曳、連接、成功鳴笛、失敗）。
  - RWD 響應式設計（確保在手機平板上拖曳順暢）。
- **Phase 4: QA 與測試 (Week 4)**
  - 邊界測試（單字過長時的縮放處理）。
  - 跨瀏覽器與跨裝置觸控測試。

### 4.2 風險控管
- **長單字溢出**：若單字過長（如 `hippopotamus`），車廂需支援動態寬度或文字自動縮放 (auto-fit)，確保不會破壞車廂的視覺比例。
- **觸控衝突**：在行動裝置上，需防止拖曳車廂時觸發瀏覽器的原生滾動（`touch-action: none`）。
