# 英文學習平台學習指南 (新手開發者與專案管理者必讀)

你好！身為一個程式新手或是專案負責人（TL），在這段時間我們一起打造這個英文學習平台（尤其是文法遊戲部分）的過程中，我們踩過了不少坑，也修復了許多錯誤。

這份指南是專門為你整理的「避坑寶典」與「除錯心法」。未來當平台要繼續擴充新功能、新遊戲時，只要回頭看看這份文件，你就能快速回想起當初我們是怎麼解決問題的，並且避免重蹈覆轍！

## 🌟 零、我們的協作模式與溝通約定 (Collaboration Agreement)

為了協助你未來順利轉職 **TPM (Technical Product Manager, 技術產品經理)** 或 **RD (Software Engineer, 軟體工程師)**，我們接下來的對話與協作將遵循以下原則：

1. **角色定位**：我是你的「線上英語遊戲設計師」兼「技術導師」。我會從產品設計與技術實作的雙重視角來協助你。
2. **溝通風格**：
   * **架構清晰**：遇到複雜的技術問題時，我會拆解成「問題現象 -> 根本原因 -> 解決方案 -> TPM/RD 學習點」的結構。
   * **白話易懂**：我會避免過度生硬的技術名詞，盡量用生活化的比喻（例如：把 API 想像成餐廳服務生）來向你解釋系統架構。
   * **聚焦核心**：我會過濾掉不需要你深究的底層程式碼細節，專注於「你需要理解的邏輯、資料流與產品決策」。
3. **問題記錄 (Bug & Solution Tracking)**：未來開發過程中遇到的所有大坑與解法，我們都會持續更新到這份文件的「第一章：經典錯誤大集合」中，作為你轉職的實戰作品集 (Portfolio) 與經驗庫。

## 🌟 第一章：我們遇過的經典錯誤大集合 (Bug Hall of Fame)

開發程式就像是在寫一篇非常嚴謹的說明文給電腦看。只要有一個小地方沒寫清楚，電腦就會罷工。以下是我們在開發平台時遇到的經典錯誤，以及背後的原因：

*(📝 **未來記錄格式範例**：我們會用以下格式把新遇到的問題記錄在這裡)*
* **【問題現象】**：(使用者看到了什麼錯誤？)
* **【根本原因】**：(程式碼或系統架構哪裡出了問題？)
* **【解決方案】**：(我們最後怎麼修復它的？)
* **【TPM/RD 學習點】**：(從產品經理或工程師的角度，我們學到了什麼教訓？如何避免下次再犯？)

### 1. 「少了一個括號的悲劇」：語法錯誤 (Syntax Error / PARSE_ERROR)
* **當時發生的事**：當我們在清理舊的程式碼（把好幾百行的教務系統搬到共用模組）時，遊戲突然完全開不起來，出現 `Expected '}' but found 'EOF'`。
* **背後原因**：在刪除一大段程式碼時，不小心漏刪了最後一個大括號 `}`。
* **TPM/RD 學習點**：刪除舊程式碼時要非常小心。可以善用編輯器（如 Cursor / VS Code）的功能，確保括號成對。作為 TPM，要知道這種「語法錯誤」通常是最低級但也最容易讓整個系統崩潰的原因。

### 2. 「幽靈變數與永遠的加載中」：未定義錯誤 (ReferenceError)
* **當時發生的事**：玩家點進遊戲，畫面一直卡在「加載中...」。
* **背後原因**：我們把舊的資料紀錄系統換成了全新的系統，但深層程式碼還在呼叫舊名字。電腦找不到舊名字就會報錯並中斷畫面更新。
* **TPM/RD 學習點**：**改名或刪除變數，必須「斬草除根」**！這就是為什麼軟體工程師常說「重構 (Refactoring) 有風險」。作為 TPM，在排程重構任務時，要預留足夠的測試時間。

### 3. 「迷路的路徑」：找不到檔案 (Can't find stylesheet to import)
* **當時發生的事**：將 CSS 樣式統一抽出成共用模組後，打包網站時突然報錯說找不到檔案。
* **背後原因**：在寫相對路徑時，`../` 代表「上一層」。不同檔案深度不同，直接「複製貼上」路徑會讓電腦迷路。
* **TPM/RD 學習點**：理解「專案目錄結構 (Directory Structure)」與「絕對/相對路徑」的概念。這也是為什麼現代前端框架喜歡設定 `@/` 作為根目錄的捷徑。

### 4. 「太過聰明的干擾選項」：題目邏輯瑕疵 (Game Design Flaw)
* **當時發生的事**：在火車重組題中，玩家拼出了文法正確的句子，系統卻判定「答錯」。
* **背後原因**：題目設計瑕疵。標準答案沒有 `always`，但干擾庫裡放了 `always`。系統是「字串嚴格比對」，所以判錯。
* **TPM/RD 學習點**：**電腦很笨，它只知道你告訴它的標準答案**。這屬於「產品邏輯/邊界條件 (Edge Case)」沒定義好的問題。TPM 在設計題庫規格時，必須確保干擾項絕對不能與正確答案產生邏輯衝突。

### 5. 「表裡不一的彈出視窗」：UI 狀態不同步 (State Confusion)
* **當時發生的事**：玩家錯了三次被扣光生命值，遊戲結束時居然顯示綠色勾勾和「答對了！」。
* **背後原因**：負責顯示解析的視窗原本只設計給「答對」使用。後來規則改成「錯三次也要跳」，卻沒傳遞「答錯」的狀態給視窗。
* **TPM/RD 學習點**：這是經典的「狀態管理 (State Management)」問題。前端工程師必須清楚每一個元件當下處於什麼狀態。TPM 在開規格時也要明確指出「不同狀態下，同一個元件該長什麼樣子」。

### 6. 「新舊系統大亂鬥」：資料格式不相容與隱藏的當機 (Data Migration & Silent Crash)
* **當時發生的事**：部分遊戲的教師儀表板卡在「載入中...」。
* **背後原因**：新版儀表板試圖讀取舊格式的資料，遇到缺少欄位（空值 `undefined`）時，JavaScript 處理文字直接崩潰，且迴圈錯誤沒有彈出警告（沉默的當機）。
* **TPM/RD 學習點**：
    * **資料庫遷移 (Data Migration) 是大魔王**：系統升級時，新系統必須能向下相容舊資料。工程師必須做好「防禦性編程 (Defensive Programming)」，例如預設值處理。
    * 作為 TPM，在規劃系統大升級時，一定要問工程師：「舊資料會怎麼處理？會不會壞掉？」

### 7. 「功能疊床架屋」：缺乏模組化思維 (Redundant Modules) 【新增】
* **當時發生的事**：我們發現原本的「現在式對話遊戲」與「自我介紹遊戲」的核心玩法幾乎一模一樣，都是給予情境提示後與 NPC 進行問答。
* **背後原因**：初期規劃時，思維侷限在「以文法單元或情境來開遊戲專案」，導致工程師可能會寫出兩套非常相似但無法共用的程式碼。
* **解決方案**：進行了「產品重構 (Product Refactoring)」，將這兩個遊戲合併成高度模組化的「互動對話角色扮演遊戲 (Dialogue Roleplay Game)」，並將變動的部分（如現在式或自我介紹的問句）抽離成 CSV 題庫。
* **TPM/RD 學習點**：
    * **高內聚，低耦合 (High Cohesion, Low Coupling)**：當發現不同功能有著相同的骨架時，TPM 應該要有警覺心，思考是否能將其合併成一個**「底層框架」**，再透過不同的資料（Data）來驅動不同的情境。這能大幅降低未來的維護成本！

### 8. 「防範防呆漏洞」：邊界條件未定與時間資料誤植 (Preventing Logic Holes) 【新增】
* **當時發生的事**：在設計「時空偵探」遊戲時，我們規劃每個時態都要問出時間/頻率提示，原本企劃只寫了 `every night, usually, always` 作為範例。
* **背後原因**：如果只寫這幾個字，工程師可能會在資料庫設計時將這些詞寫死 (Hardcode) 在遊戲核心裡，導致未來玩家在玩「過去式」關卡時，畫面還是跑出 `every night` 這種現在式的詞彙。
* **解決方案**：在規格書中明確補上「**各個時態都必須配有其專屬的時間/頻率提示詞**」，並強調這些詞必須動態從 CSV 題庫裡讀取，不得寫死。
* **TPM/RD 學習點**：
    * **資料字典與限制條件 (Data Dictionary & Constraints)**：提供開發規格時，必須嚴格定義「什麼資料只能在什麼情況下出現」。
    * **預見未來 (Foresight)**：TPM 在開規格時，要永遠假設「如果這個功能要擴充到別的單元，現在的設計會不會壞掉？」這就是確保系統「擴充性 (Scalability)」的展現。

### 9. 「單字全部不見了」還有「兩個下拉選單分不清楚」：配對遊戲 (Matching Game) 的真實案例 【新增】

這件事發生在 **同義詞／反義詞配對遊戲**（`apps/Matching-Game`）：畫面上明明有 Synonyms / Word / Antonyms 三個欄位標題，中間卻一片空白，像遊戲壞掉；同時使用者反映「下拉選單看起來像在選段落，不是選單元」。

* **【問題現象】**
    * 打開某個單元（例如 YLE-2）後，**一張單字卡都沒出來**，計時與分數像有在跑，但主內容區是空的。
    * 畫面上有 **兩個下拉選單** 排在一起，沒有清楚說明時，很容易以為「兩個都在選單元」，或把其中一個當成「選課文段落」卻不知道另一個才是單元。

* **【根本原因】（用白話說）**
    1. **檔案開頭的「隱形字」**：我們用 Excel 把題庫存成 **UTF-8 的 CSV** 時，檔案最前面常常多一個電腦才看得懂的記號，叫做 **BOM**（Byte Order Mark）。人眼看表頭還是 `Word`，程式卻讀成「某個看不見的字 + Word」，結果 **對不到欄位名稱**，整份題庫被當成 **0 筆資料**，畫面就空了。  
       另外一度還發生過：程式呼叫了「去掉 BOM」的函式，**函式卻還沒寫好**，瀏覽器直接報錯中斷，畫面也會停在半套舊 HTML（例如標題還沒改成 `YLE-2-Matching`）。
    2. **一個畫面、兩種不同意思的下拉選單**：  
        * 第一個應該是 **「單元」**（YLE-1、YLE-2…要讀哪一份教材題庫）。  
        * 第二個是 **「段落」**（只練課文裡的第幾段相關單字），只在使用 **Dictionary 題庫** 且有 **Para（段落）欄** 時才有意義。  
        若畫面上沒寫「單元／段落」，使用者會把 **底下的數字分頁鈕**（第幾「頁」遊戲）和 **段落** 搞混——其實它們是不同層級：單元 → 題庫檔；段落 → 題庫裡的 Para；數字鈕 → 依 Para 分好的遊戲頁。

* **【解決方案】**
    * **BOM**：程式在讀入 CSV 字串後先做 **`stripUtf8Bom`**，表頭每個欄位名稱也正規化時順便去掉 BOM，這樣 `Word`、`Para` 等欄位才對得準。  
    * **兩個下拉選單**：在網頁上用 **文字標籤** 寫清楚「單元」「段落」；段落選單只在 Dictionary 模式填入「全部段落」「第 N 段」，並與網址 `?para=` 同步；切換單元時盡量 **保留** 網址上的 `para`，比較符合老師「換單元但還想對齊某段」的直覺。  
    * 詳細規格與實作對照已寫入：`apps/Matching-Game/docs/Matching-Game-Rule.md`（建議閱讀 §3.4～§3.6、§6.1、§9）。

* **【TPM/RD 學習點】**
    * **資料產出要假設「Excel 會加料」**：給遊戲讀的 CSV，要當成「可能帶 BOM、可能多空行」的髒資料，程式要有防禦；驗收時除了看畫面，也要用 **一筆真實從 Excel 匯出的檔** 測一次。  
    * **一個畫面兩個選單 = 兩套產品語意**：TPM 要在規格書寫清楚「叫什麼、影響什麼網址參數、和分頁鈕差在哪」；設計 UI 時寧可字多一點，也不要讓使用者猜。  
    * **空白畫面不一定是 CSS**：若完全沒卡片，優先懷疑「資料沒載進來」或「JS 中途報錯」，請開 F12 **Console** 看紅字，比盲目調樣式快很多。

#### 9.1 補充：為什麼「大廳首頁」段落數和「連連看」裡的下拉不一樣？（YLE-2 實例）

* **你看到的狀況**：閱讀大廳裡「同反義詞連連看」只列到 **第 5 段**，進入遊戲後段落選單卻出現 **第 6 段**。
* **原因（白話）**：大廳的連結是用 **`units.json` 裡寫死的 `paragraphs: 5`** 去「迴圈產生」的；遊戲裡的段落選單則是讀 **`YLE-2-Dictionary.csv` 的 `Para` 欄**（每列可能對應第幾段，複合值取最小段號），題庫裡確實有單字落在 **第 6 段**，所以遊戲會顯示第 6 段。**兩邊參考了不同份「真相」**，數字就對不起來。
* **我們怎麼統一**：  
    1. 已把 `units.json` 裡 YLE-2 的 `paragraphs` 改成 **6**（與目前題庫一致）。  
    2. 大廳頁面改為：若有 Dictionary 檔，會 **自動抓題庫裡 Para 的最大段號**，再和 `units.json` 取較大者，**盡量與 Matching Game 同一套規則**，以後較不容易再少一段。

## 🚀 第二章：給程式小白 / 轉職 TPM 的求生法則

雖然你不一定要親自寫程式碼，但在帶領專案或與工程師合作時，掌握以下原則能幫你省下大把時間：

### 法則一：不要怕紅字，請把錯誤訊息當成「線索」
* 當網頁白畫面，或是按鈕沒反應時，請按下鍵盤上的 **F12 (或右鍵 -> 檢查)**，打開瀏覽器的「**主控台 (Console)**」。
* 如果裡面有紅色的字，那通常就是破案的關鍵（如 `ReferenceError`, `SyntaxError`）。

### 法則二：「牽一髮動全身」，永遠要全域搜尋
* 作為一個平台，很多元件和資料是共用的。如果你要求工程師「把某個功能改名」，一定要提醒：「**請記得檢查所有使用到這個變數的檔案，一併修改。**」

### 法則三：不只要測「正常情況」，還要測「各種死法」 (Edge Cases)
* 在測試遊戲時，不要只測試「一路答對到破關」。
* 軟體開發有一句話：「正常情況通常會運作，但錯誤情況最容易崩潰。」你必須刻意測試亂點、斷線、扣光愛心、重新整理等極端狀況。

### 法則四：善用「模組化」的概念 (Modularity)
* 就像玩樂高一樣，我們把「計分」、「生命值」、「儀表板」做成一塊一塊的「共用積木」。未來要做新遊戲，只要把積木插上去，就能確保全站體驗一致，減少重複開發。

## 🧠 第三章：轉職 TPM / RD 的思維升級指南

未來的開發與問題解決中，我們不再只看「程式碼怎麼寫」，而是要從更高的維度來思考：

### 1. 產品思維 (Product Thinking) - "Why are we doing this?"
* **使用者體驗 (UX) 優先**：第一時間思考「這會怎麼影響學生的學習體驗？」
* **定義邊界條件 (Edge Cases)**：好的 TPM/RD 會想到「如果使用者不按牌理出牌怎麼辦？」

### 2. 架構思維 (Architectural Thinking) - "How does this fit together?"
* **高內聚，低耦合 (High Cohesion, Low Coupling)**：各自管好自己的事。把「計分邏輯」和「畫面顯示」分開，改畫面就不會弄壞計分。
* **資料流 (Data Flow)**：培養追蹤「資料從哪裡來，去了哪裡」的能力。只要掌握資料流，遇到 Bug 就能快速縮小偵查範圍。

### 3. 溝通思維 (Communication Thinking) - "How to translate?"
* **成為工程師與使用者的橋樑**：TPM 最重要的能力是「翻譯」。把「畫面好卡」翻譯成「API 回應時間過長，需要做快取 (Cache)」。
* **定義清楚的驗收標準 (Acceptance Criteria, AC)**：給工程師（或 AI）明確的目標。例如：「我希望有一個按鈕，點擊後會顯示提示。條件是：1. 只能點 3 次，2. 點擊要有動畫，3. 點完後按鈕變灰色。」

## 🏗️ 第四章：專案架構與流程優化實戰 (Architecture & Process Optimization)

在我們將「單字與閱讀模組 (Word & Reading)」的產線自動化時，我們從 TPM 與 TL 的高階視角進行了一次深度的架構重構。這些經驗對於未來規劃任何大型資料處理或 AI 生成專案都非常關鍵。

### 1. TPM 視角：流程、風險與專案管理 (Process, Risk & Project Management)

*   **優化點 C：設立「人工審核節點」 (Human-in-the-Loop, HITL / QA Gate)**
    *   **情境**：原本的流程讓 AI 從頭到尾自動生成單字、同義詞、例句，然後直接餵給遊戲和講義。
    *   **風險**：AI 會有幻覺 (Hallucination)。如果核心的字典檔 (Dictionary) 生成了錯誤的詞性或不通順的例句，後續所有的遊戲和講義都會產出垃圾內容，這就是所謂的 **Garbage in, garbage out (GIGO)**。
    *   **解法**：在關鍵的資料生成後，強制加入「Step 3.5: 教材編輯人工審核」。確認無誤後才能放行進入下一階段。
*   **優化點 D：明確標示「自動化邊界」與「執行角色」**
    *   **情境**：SOP 文件中只寫了「動作」，看不出是人要手動做、AI 生成、還是程式腳本自動跑。
    *   **解法**：在每個步驟標題加上明確的執行標籤（如 `[AI 生成]`、`[Python 自動化]`、`[人工審核]`、`[前端渲染]`）。這能讓接手專案的人瞬間明白該步驟的執行成本與所需工具。

### 2. TL 視角：系統架構、資料流與工程 (Architecture, Data Flow & Engineering)

*   **優化點 E：確立「單一真理來源」 (Single Source of Truth, SSOT)**
    *   **情境**：原本在「同義詞遊戲」和「單字測驗」的步驟中，寫著讓 AI「生成」遊戲所需的單字資料。
    *   **風險**：我們在前面的步驟已經生成過一次單字資料了。如果讓 AI 在不同步驟重複生成相同的內容，會導致資料不一致（例如：同一個單字在講義裡的例句，跟遊戲裡的例句不一樣）。
    *   **解法**：嚴格禁止在下游任務中重新生成核心資料。下游的遊戲與測驗只能「讀取與渲染」那份唯一的字典檔 (`Dictionary.csv`)。
*   **優化點 F：防呆的「命名策略」與「絕對路徑」**
    *   **情境**：原本的原始單字檔叫 `orgword.csv`，目標單字檔叫 `word.csv`，且沒有寫明存檔路徑。
    *   **風險**：檔名太像（只差幾個字母）極易導致工程師寫錯腳本或 PM 抓錯檔案；沒有絕對路徑會讓檔案散落在專案根目錄，變成垃圾堆。
    *   **解法**：將 `word.csv` 改名為語意極度明確的 `Dictionary.csv`，與 `orgword.csv` (或 `RawWords.csv`) 做出強烈對比。同時在 SOP 中強制規定所有產出檔案的絕對儲存路徑（如 `content/vocabulary/{單元}/...`）。

### ✅ 未來專案檢查清單 (Future Project Checklist)

未來當你接手或規劃任何新的專案（尤其是包含 AI 自動化生成的專案）時，請務必拿這份清單問自己：

#### 🛡️ TPM 檢查清單 (風險與流程)
1.  **[ ] 核心資料是否有 QA Gate？**
    *   *自問*：AI 生成的資料如果出錯，會不會直接毀了使用者的體驗？如果是，我有沒有在流程中設計「人工審核 (Human-in-the-Loop)」的煞車點？
2.  **[ ] SOP 的執行角色清楚嗎？**
    *   *自問*：這份規格書交給新人，他能一眼看出哪一步要自己寫 Prompt 餵給 AI，哪一步只要點擊腳本執行嗎？

#### ⚙️ TL 檢查清單 (架構與資料流)
3.  **[ ] 是否打破了「單一真理來源 (SSOT)」？**
    *   *自問*：同樣的資料（如單字、題目、設定）有沒有在不同的地方被重複生成或重複儲存？如果有，立刻重構，讓所有系統都指向同一個源頭。
4.  **[ ] 檔名與路徑是否具備「防呆」能力？**
    *   *自問*：兩個不同用途的檔案，名字會不會太像？（例如 `data1.csv` 和 `data2.csv` 是不及格的命名）。檔案產出後，有沒有明確的專屬資料夾可以收納？
